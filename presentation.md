# 銀行帳戶交易管理系統 (Bank Account Transaction Management System)

## 專題簡報

---

### 第 1 頁：封面 (Title Slide)

**專題名稱**：銀行帳戶交易管理系統
**開發者**：Brian Wen
**日期**：2025年11月27日
**技術架構**：Vue 3 + Express.js + MySQL

---

### 第 2 頁：專案簡介 (Introduction)

- **專案概述**：
  - 本系統為一個全功能的線上銀行模擬平台。
  - 旨在提供使用者安全、便捷的帳戶管理與資金操作體驗。
  - 整合了現代化的前端介面與穩健的後端資料處理。
- **目標受眾**：
  - 一般網銀使用者（存款、轉帳、理財）。
  - 銀行系統管理員（帳戶監控、風險管理）。

---

### 第 3 頁：開發動機與目標 (Motivation & Goals)

- **開發動機**：
  - 模擬真實金融系統的運作邏輯（如：ACID 交易特性）。
  - 實踐前後端分離 (SPA) 的開發模式。
  - 解決傳統銀行介面操作繁瑣、體驗不佳的痛點。
- **專案目標**：
  - **安全性**：確保數據傳輸與儲存的安全。
  - **即時性**：交易後的餘額與記錄需即時更新。
  - **易用性**：提供直觀、美觀的操作介面。

---

### 第 4 頁：核心功能 - 用戶端 (User Features)

- **帳戶安全**：
  - **註冊/登入**：支援真實姓名驗證、密碼保護。
  - **狀態管理**：登入狀態持久化，自動逾時登出。
- **儀表板 (Dashboard)**：
  - 總資產概覽（主帳戶 + 子帳戶）。
  - 近期交易活動摘要。
  - 視覺化資產分佈圖表。

---

### 第 5 頁：核心功能 - 交易系統 (Transaction System)

- **存款 (Deposit)**：
  - 支援多種幣別或來源模擬。
  - 即時更新帳戶餘額。
- **提款 (Withdraw)**：
  - 餘額檢查機制，防止超額提款。
- **轉帳 (Transfer)**：
  - **內部轉帳**：主帳戶與子帳戶互轉。
  - **外部轉帳**：轉帳至其他用戶，支援「常用帳號」快速選取。
  - **交易備註**：紀錄轉帳用途。

---

### 第 6 頁：核心功能 - 子帳戶管理 (Sub-accounts)

- **功能亮點**：
  - 用戶可建立多個子帳戶（如：旅遊基金、儲蓄目標）。
  - 每個子帳戶擁有獨立的 ID 與餘額。
  - 自定義子帳戶名稱與代表顏色（視覺化區分）。
- **應用場景**：
  - 專款專用，協助用戶進行理財規劃。

---

### 第 7 頁：核心功能 - 管理員後台 (Admin Dashboard)

- **用戶管理**：
  - 查看所有註冊用戶列表。
  - **凍結/解凍帳戶**：針對異常帳戶進行權限控管。
- **資金控管**：
  - 管理員可手動調整用戶餘額（模擬臨櫃操作）。
- **系統監控**：
  - 查看全系統交易流水。
  - 統計系統總資產與用戶數。

---

### 第 8 頁：系統架構 (System Architecture)

- **採用前後端分離架構 (Client-Server Architecture)**
- **前端 (Frontend)**：
  - 負責畫面渲染、使用者交互、狀態管理。
  - 透過 RESTful API 與後端溝通。
- **後端 (Backend)**：
  - 負責業務邏輯處理、資料驗證、資料庫操作。
- **資料庫 (Database)**：
  - 負責資料的持久化儲存與關聯維護。

---

### 第 9 頁：技術棧詳解 - 前端 (Frontend Stack)

- **Vue 3 (Composition API)**：
  - 更靈活的邏輯複用，程式碼結構更清晰。
- **Pinia**：
  - 新一代狀態管理庫，取代 Vuex，負責管理 User Session 與帳戶資料。
- **Vue Router**：
  - 處理 SPA 的頁面路由與導航守衛 (Navigation Guards)。
- **Chart.js**：
  - 繪製資產分佈圓餅圖與交易趨勢圖。
- **Vite**：
  - 極速的開發伺服器與建置工具。

---

### 第 10 頁：技術棧詳解 - 後端 (Backend Stack)

- **Node.js & Express.js**：
  - 輕量級、高效能的 Web 伺服器框架。
  - 設計 RESTful API Endpoints (如 `/api/login`, `/api/transaction`)。
- **MySQL & mysql2**：
  - 關聯式資料庫，確保交易資料的一致性 (Consistency)。
  - 使用 Connection Pool 提升併發處理效能。
- **安全性實踐**：
  - SQL Injection 防護（使用 Parameterized Queries）。
  - CORS 策略配置。

---

### 第 11 頁：資料庫設計 (Database Schema)

- **Users (用戶表)**：
  - 儲存 ID, 帳號, 密碼, 真實姓名, 角色 (Admin/User), 狀態。
- **Sub_accounts (子帳戶表)**：
  - 關聯 User ID，儲存子帳戶名稱、餘額、顏色。
- **Transactions (交易記錄表)**：
  - 紀錄交易類型 (存款/提款/轉帳)、金額、時間、關聯帳戶。
- **Favorite_accounts (常用帳號表)**：
  - 多對多關聯，儲存用戶的常用轉帳對象。

### 第 13 頁：開發挑戰與解決方案 (Challenges & Solutions)

- **挑戰 1：資料一致性**
  - _問題_：轉帳時，扣款成功但入帳失敗怎麼辦？
  - _解法_：使用資料庫 Transaction (交易) 機制，確保 `COMMIT` 或 `ROLLBACK` 同步執行。
- **挑戰 2：前端狀態同步**
  - _問題_：交易後，頁面餘額未即時更新。
  - _解法_：利用 Pinia Store 統一管理狀態，交易成功後強制觸發 `fetchUserData` 重新拉取最新資料。
